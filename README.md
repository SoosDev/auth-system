# 認証システム (Auth System)

Node.js + TypeScript + PostgreSQL を使用した、本番品質の JWT 認証システムです。
ポートフォリオ用として、実際の SaaS で使われる認証パターンを実装しています。

## アーキテクチャ

```
クライアント (React)
       ↕ HTTP / Bearer Token
Fastify API (Node.js + TypeScript)
       ↕ SQL (Drizzle ORM + indexes)
PostgreSQL
```

### トークンモデル

| トークン | 種類 | 有効期限 | 保存場所 |
|----------|------|----------|----------|
| アクセストークン | JWT (HS256) | 15分 | クライアント（メモリ） |
| リフレッシュトークン | ランダム文字列 | 30日 | DB（ハッシュ済み） |

## フロントエンド デモ

## なぜリフレッシュトークンのローテーションが重要か

リフレッシュトークンを使用するたびに新しいトークンに置き換えます（ローテーション）。

古いトークンが再利用された場合、それはトークンが盗まれたサインです。
本システムは `previous_refresh_token_hash` カラムを使い、古いトークンの再利用を検知します。
検知時はそのユーザーの全セッションを即座に無効化し、攻撃の被害を最小化します。

## アクセストークンの即時無効化について

JWT はステートレスです。そのため、セッションを取り消しても、既に発行済みのアクセストークンは有効期限（15分）まで技術的に有効です。

本システムはこの問題を以下の方法で軽減しています：

1. アクセストークンの有効期限を短く設定（15分）
2. 認証ミドルウェアでリクエストごとにセッションの有効性をDBで確認
3. セッションが取り消されていれば即座に 401 を返す

DB チェックはリクエストごとに1回のクエリを追加します。
高トラフィック環境では Redis でキャッシュすることで最適化できます（TTL = アクセストークンの有効期限）。

## トークン保管の設計方針

リフレッシュトークンはレスポンスボディで返し、クライアントはメモリに保持します（localStorage は XSS のリスクがあるため不使用）。
httpOnly Cookie による保管も可能ですが、その場合は CSRF 対策（SameSite=Strict または CSRF トークン）が必要です。
このシステムでは、シンプルさと明確さを優先してヘッダーベースの認証を採用しました。

## セキュリティ設計の前提

**`refresh_token_hash` のユニークインデックス:** ハッシュ値は全ユーザーを通じてグローバルに一意です。衝突防止は DB レベルで強制されています（384ビットランダムトークンでの衝突は事実上不可能ですが、設計上の明示的な決定です）。

**logout-all 後のアクセストークン:** `/auth/logout-all` は全セッションの `revokedAt` を即座に更新します。`authenticate` ミドルウェアはリクエストごとにセッション有効性を DB で確認するため、呼び出し元のアクセストークンもそのリクエスト以降は無効になります。

**アクセストークンのバインディングなし:** IP・UA へのバインドは行っていません。モバイル環境での IP 変化などによる誤検知を避けるための意図的な決定です。

## セキュリティ対策

- argon2id によるパスワードハッシュ（OWASP 推奨）
- リフレッシュトークンのローテーション（再利用検知 + 全セッション失効）
- DB にはトークンのハッシュのみ保存（SHA-256、平文なし）
- セッション単位の失効（デバイスごとに管理可能）
- リクエストごとのセッション有効性確認（ミドルウェア）
- パスワード変更時に全セッション無効化（トランザクション）
- ログインへのレート制限（1分5回）
- RBAC（ロールベースアクセス制御）
- タイミング攻撃対策（存在しないメールでも argon2 を実行）
- セッションテーブルへのインデックス（user_id、refresh_token_hash）
- セキュリティイベントログ（LOGIN_FAILURE・REFRESH_TOKEN_REUSE）

## API エンドポイント

| メソッド | パス | 説明 |
|----------|------|------|
| POST | /auth/register | ユーザー登録 |
| POST | /auth/login | ログイン（アクセス＋リフレッシュトークン発行） |
| POST | /auth/refresh | トークンリフレッシュ（ローテーション） |
| POST | /auth/logout | ログアウト（現在のセッション無効化） |
| POST | /auth/logout-all | 全デバイスからログアウト（全セッション無効化） |
| GET | /me | 現在のユーザー情報 |
| GET | /sessions | アクティブなセッション一覧 |
| DELETE | /sessions/:id | 特定セッションを失効 |
| POST | /me/change-password | パスワード変更（全セッション失効） |

## ローカル実行方法

### 前提条件

- Node.js 22+
- Docker Desktop for Windows

```powershell
# リポジトリをクローン
git clone <repo-url>
cd auth-system

# 環境変数の設定
Copy-Item .env.example .env
# .env の JWT_SECRET を以下で生成した値に変更:
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"

# データベースを起動
docker-compose up -d db

# 依存パッケージのインストール
npm install

# マイグレーションとシードを実行
npm run db:migrate
npm run db:seed

# 開発サーバーを起動
npm run dev
```

API は `http://localhost:3000` で利用可能です。
フロントエンドは `cd client && npm install && npm run dev` で起動します。
